Закрывает открытый пункт SPEC §14 «авторизация сайта (SMS vs email)».

## Решения (2026-07-22, Василий)

| Вопрос | Решение | Обоснование |
|---|---|---|
| Метод входа v1 | **Email (логин) + пароль** | Паритет с muru.ru; альфа-имя на модерации у операторов |
| SMS-OTP | Отложен в **W5**, добавляется *вторым методом* на ту же сущность | Модерация альфа-имени — недели, не блокирует код |
| Капча | **Yandex SmartCaptcha** (как на muru.ru) | RU-хостинг, паритет |
| Подтверждение email | **Обязательно, double opt-in** | Без него автопривязка гостевых заказов = чтение чужих заказов |
| Телефон при регистрации | Необязателен; обязателен в «Персональных данных» и на чекауте | Паритет с muru.ru |
| Адреса доставки | **В v1**, отдельная таблица (не JSONB) | Нужны структурированные поля под СДЭК (город/ПВЗ) |
| Гостевой заказ | **Остаётся** | Не резать конверсию |
| Мин. длина пароля | **8** (на muru.ru — 6, не копируем) | Дёшево, снимает основную массу брутфорса |

## Продуктовый скоуп v1

Страницы storefront (референс — скриншоты muru.ru, сессия 41):
- `/register` — ФИО, email, телефон (опц.), пароль ×2, SmartCaptcha, согласие 152-ФЗ
- `/login` — email + пароль; `/password/reset` — запрос и установка нового
- `/account` — сводка: личные данные, плитки «Избранное» / «Заказы», блок «Активные заказы»
- `/account/personal` — контактные данные + смена пароля
- `/account/orders` — активные и завершённые, карточка с №, статусом, миниатюрами, прогрессом
- `/account/favorites` — избранное
- `/account/addresses` — сохранённые адреса, адрес по умолчанию

## Модель данных (предварительно — финализируется по итогам W0)

Миграция **031**. Не является чисто аддитивной → **staging-first обязателен**.

- `customers` — id, email UNIQUE (lowercase), password_hash (bcrypt), full_name,
  phone NULL, **phone_verified_at NULL**, **email_verified_at NULL**, telegram_id NULL,
  is_active, created_at, last_login_at
  *(поля phone_verified_at / telegram_id заложены под W5 и склейку TG+web по SPEC §8 —
  чтобы SMS-OTP лёг дополнительным методом, а не переписыванием личностей)*
- `customer_addresses` — id, customer_id FK, label, city, cdek_city_code, address,
  is_default, created_at
- `customer_auth_tokens` — id, customer_id FK, kind ('email_verify' | 'password_reset'),
  token_hash, expires_at, used_at NULL
- `orders` — ADD `customer_id` NULL FK, `customer_email` NULL, `customer_phone` NULL
  (нормализованный E.164) + бэкфилл. **Без этого раздел «Заказы» в ЛК показать нечего:**
  на 2026-07-22 в схеме нет ни email, ни телефона у заказа, только `telegram_user_id` (NULL) и `channel`
- `favorites` — DROP NOT NULL на `telegram_user_id`, ADD `customer_id` NULL FK,
  партиальные UNIQUE-индексы на каждую из двух веток

**Инвариант мини-аппа:** все изменения `orders` / `favorites` — только аддитивные;
существующие публичные эндпоинты и их ответы не меняются.

## Безопасность

- `CUSTOMER_JWT_SECRET` — **отдельный** от `ADMIN_JWT_SECRET`, никогда не переиспользуется
- Сессия: httpOnly + Secure + SameSite cookie через Next.js BFF; refresh 180 дней
- SmartCaptcha с серверной валидацией: регистрация, сброс пароля, повторная отправка письма,
  вход после 3 неудач
- Одинаковый текст ошибки при неверном email и неверном пароле (нет user enumeration)
- Токены (verify / reset): одноразовые, TTL 60 мин, в БД только хеш
- Rate limit: вход, регистрация, отправка писем — по email и по IP
- Согласие 152-ФЗ при регистрации — по образцу `orders.consent_*` (версия + timestamp)

## Внешние зависимости

| Зависимость | Статус | Срок | Блокирует |
|---|---|---|---|
| SMTP на домене `muru.ru` (ящик + SPF/DKIM/DMARC) | **не подтверждён** | часы-сутки | W1 (сброс пароля, verify) |
| Yandex SmartCaptcha (ключи) | не заведён | часы | W1 |
| Альфа-имя у операторов (sms.ru) | **подан 2026-07-22** | недели | только W5 |

**Мина в существующем коде:** `backend/src/services/email.service.ts` при неконфигурированном SMTP
**молча возвращается без ошибки**, а лог пишет только при `NODE_ENV !== 'production'`.
Для auth-флоу это недопустимо. В W1: SMTP становится обязательным при включённом модуле ЛК,
проверка транспорта при старте, ошибка отправки — видимая пользователю и в логах.

**Экономика SMS (проверено 2026-07-22, sms.ru):** абонплата за альфа-имя — Мегафон 2550 ₽/мес,
МТС 2040 ₽/мес, Билайн 2080 ₽/мес, Т2 0 ₽ (2450 ₽ с шаблонами) ⇒ фикс ≈ 6 670 ₽/мес.
За SMS: МТС 5,64 ₽ / Т2 7,56 ₽. Бесплатное общее имя непригодно для OTP (задержка 5–15 мин).
Авторизация звонком — 40 коп., без альфа-имени; у sms.ru старая схема (входящий звонок,
код = последние 4 цифры) **выведена из обращения**, новая требует исходящего звонка
пользователя на 8-800 → на десктопе UX плохой. Вывод: SMS основной канал, звонок — fallback/dev.
